# 建房订单业务流程与测试说明

## 1. 前端启动方式

### 1.1 环境要求

- Node.js 18+
- npm 9+

### 1.2 安装依赖

```bash
npm install
```

### 1.3 启动前端

```bash
npm run dev
```

默认启动地址：

```text
http://localhost:3000
```

说明：

- 开发环境下前端会通过 Vite 代理把 `/api` 请求转发到 `http://116.198.38.236:8080`
- 如果后端地址变更，需要同步修改 `vite.config.js` 和 `src/api/request.js`

### 1.4 打包验证

```bash
npm run build
```

## 2. 角色与测试入口

本项目至少涉及以下两个主要测试视角：

- 用户端：用户下单、查看订单、支付账单、提交选配变更、查看施工进度
- 后台管理端：上传合同、派单、开启施工、同步节点金额、审核选配变更、查看账单

## 3. 整体业务主线

完整主线如下：

1. 用户下单
2. 用户支付定金
3. 后台上传合同
4. 后台派单，服务商接单
5. 后台确认开工节点金额并开启施工
6. 首节点按当前后端规则跳过支付，进入施工流程
7. 后续节点按账单支付推进施工流程
8. 施工过程中允许继续发生选配变更
9. 补价走账单，减项走审核后退款或直接调总价
10. 节点全部完成后订单完结

## 4. 订单阶段说明

订单状态按当前业务可理解为：

- `0 未派单`
- `1 派单中`
- `2 已派单`
- `3 施工中`
- `4 已完成`
- `5 已取消`

当前门禁规则：

- `0/1/2/3`：允许按业务规则继续处理
- `4/5`：仅查看，不再允许继续提交选配变更、派单或同步节点金额

## 5. 用户端业务流

### 5.1 下单与定金

用户下单时可以选择：

- 主体产品
- 选配产品

提交订单后，先进入定金支付流程。

### 5.2 开工前用户修改选配

开工前允许用户在用户中心修改选配。

处理规则：

- 纯增加：直接更新选配和订单总价，并生成选配补价账单
- 减少：提交后台审核；若当前尚未支付任何施工节点款，审核通过后直接调整总价，不走真实退款
- 平替/混合调整：提交后台审核；若审核结果为补差价，则按后端规则处理补价；若为减价且尚未支付任何施工节点款，则直接调整总价，不走真实退款

### 5.3 施工后用户修改选配

施工开始后，用户仍可继续提交选配变更。

处理规则：

- 纯增加：直接生成选配补价账单
- 减少：提交后台审核；审核通过后按当前订单最近一次已支付节点款流水处理退款
- 平替/混合调整：按净差额处理
  - 需要补价：生成补价账单
  - 需要退款：按当前订单最近一次已支付节点款流水处理退款

### 5.4 用户端账单支付

用户需要关注两类账单：

- 施工节点账单
- 选配补价账单

支付入口统一在“账单支付”中查看和处理。

### 5.5 用户端施工进度

施工中用户可查看：

- 当前施工节点
- 节点状态
- 待支付节点账单
- 上传/审核推进情况

## 6. 后台管理端业务流

### 6.1 合同上传

用户支付定金后，后台上传合同，之后进入派单阶段。

### 6.2 派单与接单

后台进行：

- 施工服务商派单
- 材料服务商派单

当相关服务商接单完成后，可进入开工准备。

### 6.3 开启施工与节点金额

后台在开工前确认节点金额方案。

规则说明：

- 首笔建房定金可单独确认
- 开启施工后，系统按后端规则同步未支付节点金额
- 已支付节点金额不允许再修改

### 6.4 施工流程推进

施工中按节点循环推进：

1. 当前节点推进
2. 产生对应账单
3. 用户支付
4. 服务商上传
5. 后台审核
6. 用户确认
7. 进入下一节点

### 6.5 后台审核选配变更

后台对用户提交的选配变更只处理最新有效记录。

审核处理方式：

- 直接通过
- 生成补价账单
- 发起退款

退款时使用规则：

- 若当前订单尚未支付任何施工节点款，则不走真实退款，直接调整总价
- 若已支付过施工节点款，则按当前订单最近一次已支付节点款流水发起退款

## 7. 推荐测试场景

### 7.1 基础主流程

建议至少完整走一次：

1. 用户下单
2. 用户支付定金
3. 后台上传合同
4. 后台派单并等待接单
5. 后台确认开工节点金额并开启施工
6. 用户支付当前节点账单
7. 推进施工流程直到进入下一节点

### 7.2 开工前选配变更

建议覆盖：

- 开工前纯增加选配
- 开工前减少选配
- 开工前平替选配

重点关注：

- 总价是否变化正确
- 是否生成正确账单
- 减项在未支付节点款时是否只是调总价而非真实退款

### 7.3 施工后选配变更

建议覆盖：

- 已支付过至少一笔节点款后，再增加选配
- 已支付过至少一笔节点款后，再减少选配
- 已支付过至少一笔节点款后，做同类平替

重点关注：

- 补价是否进入账单支付
- 退款是否由后台审核后发起
- 退款是否关联当前订单最近一次已支付节点款流水

### 7.4 施工金额同步

建议覆盖：

- 开工时初始化节点金额
- 施工中重新同步未支付节点金额
- 已支付节点是否保持锁定

### 7.5 终态门禁

建议覆盖：

- 已完成订单是否只读
- 已取消订单是否只读
- 完成/取消后是否仍能提交选配、继续派单、继续同步节点金额

## 8. 当前测试注意事项

当前需要特别注意一个已知业务点：

- 如果订单从“有选配”变更为“完全没有选配”，需要以后端最终修复结果为准再做最终验收

除该场景外，其余主链路可按本文档进行联调和测试。
